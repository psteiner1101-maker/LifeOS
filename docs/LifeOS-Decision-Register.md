| D1 | Core architecture | Three layers: Shared Platform Services, Universal Productivity Objects, Specialized Domain Records | Avoids both over-generalization (forcing Bills/Assignments into generic Tasks) and module-based duplication | Core-Domain-Architecture-v2.md | **Approved** | A record type genuinely needing a fourth layer type |
| D2 | Specialized vs. universal records | Layer 3 records may have real structured fields but must consume Layer 1 services and are not required to link to a Layer 2 object | Prevents fake-schema-via-tags while keeping one shared foundation | Core-Domain-Architecture-v2.md | **Approved** | — |
| D3 | Spaces vs. Projects vs. Folders vs. Tags | Space = lasting area; Project = temporary goal-bound container; Folder = pure visual organization; Tag = lightweight label | Prevents blurring a permanent concept with a temporary one | Core-Domain-Architecture-v2.md; refined in Architecture-Final-Review-and-Amendments.md Parts 2–3 | **Approved** | — |
| D4 | Event vs. Appointment | Event owns all scheduling; Appointment owns appointment-specific info and links to exactly one Event | Single source of truth for scheduled time | Core-Domain-Architecture-v2.md; schema in Database-Schema-Design.md Part 9 | **Approved** | — |
| D5 | Bill vs. Bill Occurrence | Bill is the recurring definition; Bill Occurrence is each billing-period instance, generated on a rolling 90-day window, history never rewritten | Preserves historical payment accuracy through recurrence changes | Core-Domain-Architecture-v2.md; Database-Schema-Design.md Part 8 | **Approved** | — |
| D6 | Assignment statuses | Five states: Not Started, In Progress, Completed, Submitted, Graded | Completing work and submitting it are different real-world events | Database-Schema-Final-Audit-and-Simplicity-Amendment.md (UX corrections doc) | **Approved** | — |

| D8 | Quick Capture confirmation | Every AI suggestion requires explicit user confirmation before becoming a structured record; Reminder is a fallback type, never a default | Prevents unreviewed AI writes; matches the most-specific-type-first rule | User-Experience-and-Wireframes.md; Core-Domain-Architecture-v2.md §8 | **Approved** | — |
| D9 | Privacy levels | Three levels: Standard, Sensitive, Restricted, layered on top of (not replacing) auth/RLS/encryption | Gives graded protection without a full permissions engine | Core-Domain-Architecture-v2.md §14.2 | **Approved** | — |
| D10 | Restricted-record behavior | Visibility governed by current, deliberate user intent — not a session boundary or a historical "viewed" timestamp | "This session" was an implementation detail, not a meaningful trigger | Architecture-Final-Review-and-Amendments.md Part 5 & Part 11.1 | **Approved** | — |
| D11 | Archive vs. Trash | Archiving a Space never rewrites contained records (derived inactivity only); Trash/Restore is a separate flow for individually deleted records | Reopening a Space must cost zero row rewrites | Core-Domain-Architecture-v2.md; Architecture-Final-Review-and-Amendments.md Part 11.2 | **Approved** | — |
| D12 | Search approach | Exact match > trigram (titles/names) > full-text (long content); no general fuzzy/typo tolerance promised | Corrects an earlier overstated claim about full-text search's capabilities | Database-Schema-Final-Audit-and-Simplicity-Amendment.md Part 5 | **Approved** | — |
| D13 | Reminder `next_fire_at` | Each reminder stores a materialized, recalculated fire timestamp rather than being computed dynamically each scheduler pass | Simple, reliable, one-query scheduler; avoids fragile multi-table joins at fire time | Database-Schema-Final-Audit-and-Simplicity-Amendment.md Part 6 | **Approved** | — |
| D14 | Bill reminder ownership | Option C — Bill definition stores default rules; each generated Occurrence gets its own actual scheduled reminder instance(s) | Supports per-occurrence exceptions and stops reminding once paid, without losing the rule's natural home | Database-Schema-Final-Audit-and-Simplicity-Amendment.md Part 7 | **Approved** | — |
| D15 | Appointment reminder ownership | Reminders live solely on the linked Event; Appointment Detail reads/writes the same rows | One reminder system per appointment, no duplicate-notification risk | Database-Schema-Final-Audit-and-Simplicity-Amendment.md Part 8; reaffirmed Part 16 | **Approved** | — |
| D16 | Polymorphic references | `record_type` + `record_id` pattern for cross-cutting tables, chosen over a shared records registry | Simpler write path and easier beginner maintenance; worst failure (orphaned row) is milder than a registry desync | Database-Schema-Design.md Part 5; re-evaluated and reaffirmed in Database-Schema-Final-Audit-and-Simplicity-Amendment.md Part 3 | **Approved** | Team/scale growth large enough that registry-level integrity becomes worth its cost |
| D17 | Controlled server-side writes | All polymorphic writes (tags, attachments, relationships, reminders, daily_priorities, Inbox conversion refs) must go through a protected server-side path; RLS alone is explicitly insufficient | Ordinary Supabase clients could otherwise attempt direct writes; RLS can't validate a polymorphic target lives in the right table | Database-Schema-Final-Audit-and-Simplicity-Amendment.md Part 16 (Final Security Amendment) | **Approved** | — |
| D18 | Workspace structure | `workspaces` + `workspace_members` created at sign-up now (one owner row), fully hidden from the v1 UI | Avoids a far larger future migration to retrofit multi-user ownership | Database-Schema-Design.md Part 4; reaffirmed Database-Schema-Final-Audit-and-Simplicity-Amendment.md Part 9 | **Approved** | Real multi-user/household collaboration being built |
| D19 | Projects and Folders in v1 | Remain fully in the schema; soft-hidden in the interface (no onboarding/nav/default-form presence; discoverable inside Space Detail) | New users should learn exactly one organizational concept: Space | Database-Schema-Final-Audit-and-Simplicity-Amendment.md Part 2 & Part 15 (Decision 2); reaffirmed Part 16 | **Approved** | — |

## Amendment and Final-Review Decisions (approved; authoritative full text in Private-Hosting-and-Two-Person-Access-Amendment.md and its Approval Record)

| ID | Decision | Status |
|---|---|---|
| C1 | Automatic 30-day Trash expiration is a system action through the protected deletion service; the Workspace-Owner-only rule governs manual permanent deletion of shared records | **Approved** |
| C2 | Archiving, reopening, and Private↔Shared visibility changes are Space-Owner-only management actions | **Approved** |
| P6-A | A removed Member's private records remain preserved and invisible — never transferred, exposed, or auto-deleted | **Approved** |
| P6-B | Zero leakage to the Workspace Owner: no counts, titles, previews, storage details, or hints of the Member's private records | **Approved** |
| P6-C | Space-level Private/Shared visibility is the primary sharing model of LifeOS | **Approved** |
| P6-D | On removal: Member-owned Shared Spaces transfer to the Workspace Owner (audit-recorded, records intact); Member-owned Private Spaces preserved invisibly per P6-A; Member-created records in Shared Spaces retained with original attribution | **Approved** |
| P7-A | Standard Supabase Auth email flow for invitations and password resets in v1; no custom email system unless a future requirement demands it | **Approved** |
| P8-A | Cross-visibility links allowed; the private end is completely invisible to the other member — no titles, previews, counts, search results, metadata, or hints; the relationship behaves as though it does not exist for anyone without access | **Approved** |
| P8-B | Structural references (e.g., Class→Instructor, Bill→Payee, Appointment→Provider) on Shared records may reference only records visible to both members; the interface never offers invalid selections; no hidden references on Shared records | **Approved** |
| P8-C | An Appointment and its Event always belong to the same Space with identical visibility; moving one automatically moves the other; an Appointment never exists without its Event (deleting an Appointment while keeping its Event unpairs that Event into a standalone Event, preserving D4's deletion-choice flow — interpretation recorded in the final completion audit) | **Approved** |
| P10-A | Today's Priority is personal per member, even for shared records; no shared household priority list in v1 | **Approved** |
| P10-B | Inbox captures are visible only to their author; visibility of the resulting record is determined at conversion by the destination Space | **Approved** |
| F6 | Attaching a File to a Shared record is a deliberate sharing action for that File; intuitive, no added confirmation dialog unless future testing proves one necessary | **Approved** |
| F9 | Mandatory terminology: Workspace Owner, Space Owner, Private Space, Shared Space, Privacy Level, Privately Hosted; bare "Owner"/"private"/"shared" avoided wherever ambiguous | **Approved** |
| ACR | Architecture Freeze: planning complete; any architectural change requires an approved Architecture Change Request before any approved document is modified | **Approved / governing** |
